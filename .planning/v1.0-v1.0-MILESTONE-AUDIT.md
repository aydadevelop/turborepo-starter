## Partial / unsatisfied requirements

- **BOOK-04: Customer and operator receive booking confirmation or cancellation notifications triggered by booking lifecycle events** (Phase 11)
  - **Current state:** the typed event path is live and customer in-app notifications are delivered.
  - **Gap:** `packages/notifications/src/events-bridge.ts` resolves only `booking.customerUserId` for `booking:confirmed` and `booking:cancelled`; no operator recipient lookup exists.
  - **Effect:** customers receive notifications, operators do not.

## Cross-phase integration findings

### Verified working live paths

- **Operator listing management -> publish/unpublish** works through `apps/web` -> `orpc.organization.*` -> `packages/organization` -> `packages/catalog` workflows.
- **Public browse/detail -> quote/availability -> booking request** works through the live storefront and booking surfaces, with server-side publication/org resolution from `listingId`.
- **Provider webhook -> payment reconciliation -> booking/payment state** works on the production ingress path through `reconcilePaymentWebhook(...)`.
- **Customer booking/support history -> support thread reply** works through `/dashboard/bookings` and `/dashboard/support/[ticketId]`.

### Critical finding

- **Booking lifecycle notifications only reach customers**
  - `packages/booking/src/booking-service.ts` and `packages/booking/src/cancellation/cancellation-workflow.ts` emit the expected typed events.
  - `apps/server/src/bootstrap.ts` correctly registers the notifications bridge.
  - `packages/notifications/src/events-bridge.ts` only resolves `booking.customerUserId`, so operator recipients are never enqueued.
