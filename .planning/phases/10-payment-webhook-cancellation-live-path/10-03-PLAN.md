  <behavior>
    Write failing workflow tests first, then make them pass.

    RED coverage to add before implementation:
    - a `requested` cancellation request with `refundAmountCents > 0` loads its stored snapshot, executes `refundPayment(...)`, marks the refund row `processed`, stores the external refund id, updates the booking/request to applied/cancelled, and emits `booking:cancelled`
    - a zero-refund request skips provider execution but still updates request/booking state correctly
    - if a downstream step fails after the refund provider call, the compensation path marks the refund row `rejected` instead of leaving an invalid or made-up status
    - changing booking dates or policy data after the request was created does **not** change the applied refund amount (stored snapshot stays authoritative)

    GREEN implementation requirements:
    1. Add `@my-app/payment` as a direct dependency of `packages/disputes`.
    2. Replace the current apply-time policy recalculation flow with a workflow input centered on applying an existing request (for example: `requestId`, `organizationId`, `appliedByUserId`).
    3. Load the persisted `bookingCancellationRequest` row inside the workflow and treat `refundAmountCents`, `currency`, `reason`, and `reasonCode` from that row as the apply-time authority.
    4. If `refundAmountCents > 0`, load the captured payment attempt and active org payment config needed for refund execution, resolve the payment provider from the new registry, and call `refundPayment(...)`.
    5. Persist the refund row as `processed` with the returned external refund id when provider execution succeeds.
    6. Update the cancellation request to `applied` and the booking to `cancelled` + `refundAmountCents` within the workflow path.
    7. Emit `booking:cancelled` only after the workflow succeeds.
    8. Compensation must use existing enum-valid statuses (`rejected`, not invented rollback states) when unwinding a partially completed refund step.

    Keep the workflow factory pattern (`processCancellationWorkflow(db)`) and do not move provider execution into API handlers.

    TDD commits:
    - `test(10-03): add failing disputes cancellation workflow coverage`
    - `feat(10-03): execute snapshot-backed refunds in disputes workflow`
  </behavior>

  </implementation>
