  <behavior>
    Write failing tests first for the live path, then make them pass.

    RED coverage to add before implementation:
    - `apps/server/src/__tests__/payment-webhook.test.ts`
      - successful `pay` webhook with a real `endpointId` on the request URL updates booking payment state through the production route instead of stubbing `adapter.processWebhook()`
      - missing `endpointId` returns HTTP 400 with a clear error body instead of pretending success
      - duplicate webhook still returns HTTP 200 `{ code: 0 }` while the payment domain reports idempotent handling internally
    - `packages/api/src/__tests__/cloudpayments.test.ts`
      - invalid HMAC header is rejected
      - valid computed HMAC header is accepted
      - Basic Auth still works as an alternative auth path
    - `packages/payment/src/__tests__/payment-service.test.ts`
      - first successful processed webhook promotes `organizationPaymentConfig.validationStatus` to `validated`, sets `validatedAt`, and flips `isActive` to `true`
      - duplicate webhook remains idempotent and does not create a second event row or payment attempt

    GREEN implementation requirements:
    1. Keep `apps/server/src/routes/payment-webhook.ts` as the production callback URL.
    2. In `packages/api/src/handlers/internal/server-routes.ts`, keep provider lookup, webhook-type validation, auth, and body parsing in the adapter layer.
    3. Resolve `endpointId` from the live request in this order:
       - `new URL(request.url).searchParams.get("endpointId")`
       - `request.headers.get("x-endpoint-id")`
       If neither exists, return HTTP 400 with a useful error body.
    4. After auth + parsing, call `reconcilePaymentWebhook(endpointId, input.webhookType, payload, db)` directly. Do **not** call `adapter.processWebhook()` on the production ingress path anymore.
    5. In `packages/payment/src/payment-service.ts`, promote the matching org payment config to `validationStatus: "validated"`, `validatedAt: now`, `isActive: true` when a known-endpoint webhook is processed successfully for the first time.
    6. Preserve current booking/payment mutation behavior for `pay`, `confirm`, `fail`, and `refund`, plus persistence-based idempotency via `paymentWebhookEvent` + `bookingPaymentAttempt`.
    7. In the CloudPayments adapter, verify the request using either valid Basic Auth or a computed SHA-256 HMAC compared against `X-Content-HMAC` or `Content-HMAC`; header presence by itself is not enough.
    8. Use `request.clone()` (or an equivalent non-destructive read) so HMAC verification does not consume the body before `parseWebhookBody()` runs.

    Keep the HTTP response contract stable for CloudPayments:
    - success path returns `{ code: 0 }`
    - processing failure returns `{ code: 1 }`
    Additional diagnostic fields are allowed only if they do not break the existing route contract.

    TDD commits:
    - `test(10-01): add failing live webhook ingress coverage`
    - `feat(10-01): route production webhooks through payment reconciliation`
  </behavior>
  <implementation>
    Keep separation of concerns strict:
    - Hono route stays a thin raw-request ingress surface
    - adapter owns provider auth/body parsing only
    - payment package owns booking/payment/config state changes

    Do not add a second public webhook endpoint and do not move provider auth logic into `packages/payment`. The point of this plan is to connect the existing live ingress to the existing domain logic without collapsing the boundary.
  </implementation>
