  <behavior>
    Change the public booking-intake contract so the browser sends only booking-intake data:
    - `listingId`
    - `startsAt`
    - `endsAt`
    - optional passengers/contact/timezone/notes/specialRequests/currency

    Remove `organizationId` and `publicationId` from both the oRPC input schema and `CreateBookingInput`.

    Then make `createBooking(input, db)` do the following in order:
    1. Look up the active marketplace publication for `input.listingId` by joining `listing` and `listingPublication`.
    2. If no active platform-marketplace publication exists, throw `Error("NOT_FOUND")`.
    3. If the listing org and publication org do not match, throw `Error("PUBLICATION_ORG_MISMATCH")`.
    4. Use the resolved values for `organizationId`, `publicationId`, and `merchantOrganizationId`.
    5. Preserve the existing slot-check and quote-calculation behavior.
    6. Insert the booking row with the resolved organization/publication context.

    RED tests to add before implementation:
    - success case: `createBooking` resolves the active publication/org from `listingId` and persists those values on the booking row
    - unpublished listing case: throws `NOT_FOUND` when the listing has no active platform-marketplace publication
    - inconsistent linkage case: throws `PUBLICATION_ORG_MISMATCH` when `listing.organizationId !== listingPublication.organizationId`
    - existing slot/pricing cases still pass (`SLOT_UNAVAILABLE`, `NO_PRICING_PROFILE`)

    Transport alignment after GREEN:
    - `packages/api-contract/src/routers/booking.ts` create input drops `organizationId` + `publicationId`
    - `packages/api/src/handlers/booking.ts` protected create handler passes only the new input fields plus session-derived `customerUserId` / `createdByUserId`
    - map `PUBLICATION_ORG_MISMATCH` to `ORPCError("PRECONDITION_FAILED")`
    - map `NOT_FOUND` to `ORPCError("NOT_FOUND", { message: "Listing is not bookable" })`

    TDD commits:
    - `test(09-02): add failing tests for server-trusted booking context`
    - `feat(09-02): resolve booking publication context server-side`
  </behavior>
  <implementation>
    Keep the owning-layer split intact:
    - contract shape in `packages/api-contract`
    - handler translation in `packages/api`
    - publication/org validation in `packages/booking`

    Do not add a second helper route just to expose publication context to the browser. The whole point of this plan is to keep that context server-trusted.

    Preserve these existing behavior guarantees from Phase 05:
    - booking creation is still protected and derives the customer id from the session
    - `assertSlotAvailable` remains the slot-safety gate
    - `calculateQuote` remains the pricing source of truth
    - inserted bookings still record `source: "web"`
  </implementation>
